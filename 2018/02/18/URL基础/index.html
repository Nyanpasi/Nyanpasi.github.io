URL基础

Posted on 2018-02-18 | 该文章被访问 次

URL的一般格式

<scheme>://<user>:<password>@<host>:<port>/<path>;<params>?<query>#<frag>

组件 含义 说明
scheme 方案 指定获取资源时使用的协议
user 用户 某些方案访问资源时需要的用户名
password 密码 用户名对应的密码
host 主机 资源宿主服务器的主机名或点分ip地址
port 端口 资源宿主服务器正在监听的端口号, 很多方案都指定了默认的端口号,
path 路径 服务器上资源的本地名
params 参数 某些方案用这个组件来指定输入参数, 形式key=value,分隔符;
query 查询 某些方案用这个组件来指定查询参数, 形式?key=value,分隔符&
frag 片段 一小片或一部分资源的名字(fragment), 仅在本地浏览器生效

方案(scheme)

  1. 大小写无关
  2. 必须以字母作为开头

主机和端口(host&port)

  1. 主机可以用主机名或者ip地址两种形式表示
  2. 在不指定端口时, 根据方案使用默认端口

用户名和密码(user&password)

  1. 这两个组件的需要与否取决于服务器, 例如FTP服务器通常需求这个组件
  2. 在需要这个组件而没有提供时, 通常会插入一个默认的用户和密码

路径(path)

  1. 路径使用/划分为多个路径段, 每个路径段都可以有自己的参数组件

参数(params)

  1. 使用;与其他组件以及互相之间进行分隔
  2. 提供访问资源所需的所有附加信息,

示例：https://example.com/path;type=1;date=2

查询(query)

  1. 使用?与其他组件进行分隔, 使用&互相之间进行分隔.
  2. 查询意在缩小请求资源类型范围.

示例：https://example.com/path?item=mirror&color=bule

片段(frag)

  1. 片段使用#与其他组件进行分隔
  2. 片段仅在html文档使用了锚点时有效
  3. 片段仅仅在本地生效, 被浏览器所处理
  4. 浏览器在请求到整个资源后, 再通过片段来定位到指定的一段

安全字符集&编码机制&字符限制

安全字符集:

各种协议都可能有自己的危险字符, 而作为通用资源定位的URL则必须避开所有的这些字符, 因此, URL能使用的字符集比US-ASCII还要小得多, 能被安全使用的字符集称之为安全字符集

编码机制:

为了能用安全字符集来表示更多的字符, 通过转义表示法来表示不安全的字符, 这样, 出现在URL中的字符都是安全的, 又能表示更多的信息, 这种转义表示法包含一个百分号, 后面跟着两个字符来表示字符ASCII码的十六进制数,

例如, 波浪号~的ASCII码为126(0x7E), 所以波浪号在出现在URL中应当被替换为%7E, 同理, 空格被替换为%20, 百分号本身则被替换为%25

显然, 这种转义表示法对于在ASCII中的所有字符都可以简单地应用(这意味这你可以转义一些本来就很安全的字符, 一样不会引起误解)

但是对于不在ASCII字符集中的字符, 编码的机制就复杂的多, 也没有统一的标准, 但通常会在URL中附加相关信息.

字符限制：

  • 本身含有不安全因素的字符出现在URL中, 必须进行转义
  • 一些保留字符, 如果不是用于保留用途之外的场合时, 必须对其进行转义

例如.这个保留在路径组件中使用的符号, 如果用在URL中而不表示其路径组件中的含义时, 就需要转义,
常见的保留字符包括. .. / # ? ; : $ + @ & = { } | \ ^ ~ [ ] < > "
另外还有一些ASCII中不可打印区间内的字符, URL不可使用

URL快捷方式

URL的表示方式有两种：绝对URL(Absolute URL), 和相对URL(Relative URL)

绝对URL包含访问资源所需的全部信息, 包含URL所有组件

相对URL则只是完整URL的一部分, 想得到完整的URL, 必须相对另一个基础URL(Base URL)进行解析

基础URL的提供方式

  • 使用HTML标记<BASE>显式提供
  • 使用当前URL

基础URL + 相对URL = 绝对URL
