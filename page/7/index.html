Redis基础

Posted on 2019-01-25

基本数据类型

Redis的键为字符串, 而值有5种基本数据类型:

  1. 字符串(String)
  2. 列表(List)
  3. 字典(Hash)
  4. 集合(Set)
  5. 有序集合(Sorted Set)

字符串

Redis字符串为动态字符串, 采用预分配冗余空间的方式来减少内存的频繁分配.

当字符串长度小于1M时, 扩容都是加倍现有的空间, 如果超过1M, 扩容时一次只会多扩1M的空间.

需要注意的是字符串最大长度为512M.

另外, 字符串的索引从0开始。

列表(List)

Redis的列表为双向链表, 首尾插入和删除较快, 而定位操作慢.

数据较少时, 使用连续内存存储, 这个结构是ziplist, 数据较多时, 使用多个链接起来的ziplist组成的quicklist.

使用列表可以方便地实现队列或者栈.

字典(hash)

一个无序字典, 字典的值只允许是字符串. 解决键冲突的方式为拉链法.

为了让负载因子维持在一个合理的范围内, 在数据增加或减少到一定程度后对其进行响应的扩展或收缩.

渐进式rehash

扩展或收缩哈希表需要将旧的hashtable里面的所有键值对rehash到新的hashtable里面

如果键值对数量很多时, 一次性rehash引起的庞大的计算量导致服务器在一段时间内停止服务.

因此, rehash动作并不是一次性、集中式地完成, 而是分多次、渐进式地完成的.

集合(set)

一个值为null的字典.

有序集合(zset)

每个元素都关联一个double类型的分数, redis通过分数进行集合成员的排序

键与分数之间的关系是用字典来实现的, 即分数作为元素的值, 但因为这个值只能是一个double类型的分数, 所以我们称之为set, 而非dict.

查询某个元素的score, 直接通过dict查询, 时间复杂度为O(1)

跳跃列表(skiplist)

zset内部排序功能是通过跳跃列表实现,

跳跃列表本质上是一种查找结构, 用于解决查找问题.

具体的实现是一个多层链表, 分为不同的层级, 相同层级的元素都使用指针进行连接.

因此某些节点同时归属于不同层级, 该节点也就包含多个后向指针.

每个节点至少包含一个前向指针, 并包含至少一个(可能多个)后向指针, 具体取决于该节点究竟同时在多少层.

  1. 最底层即L0层, 包含所有的元素
  2. 次底层即L1层, 每个元素有50%的机会进入这一层
  3. 第三层即L2层, 每个元素有25%的机会进入这一层
  4. 以此类推, 最多有32层.

显然, 最底层是一个双向链表, 而上层则不然.

每次的搜索过程从现存的最高层开始, 向下逐级搜索.

由于所有节点的晋升依赖于概率, 无需维持各层之间的数量约束关系, 因此插入效率比平衡树类型的索引要高.

而查找的平均时间复杂度是O(logn), 与平衡树类型相当.

查询元素名次, 通过分数反查元素范围等, 因为要使用skiplist, 所以时间复杂度为O(logn+M), M是要返回的元素的个数.

Redis淘汰策略

六种淘汰机制, 使用maxmemory-policy设置项进行设置

maxmemory-policy 执行策略
noeviction 不进行淘汰
allkeys-lru 对所有key按照LRU原则进行淘汰
allkeys-random 对所有key进行随机淘汰
volatile-lru 对易失key按照LRU原则进行淘汰
volatile-random 对易失key进行随机淘汰
volatile-ttl 对易失key按照生存时间最少进行淘汰

注:

  1. LRU(Least Recently Used) 最近最少使用
  2. TTL(Time To Live) 生存时间
  3. volatile: 设置了超时的key被视为volatile, 即易失的

Redis 持久化机制

  1. RDB机制, 指定时间间隔内将内存中的快照集快照写入磁盘. 默认为dump.rdb
  2. AOF机制, Append Only File, 每个写命令都通过write追加到文件中. 默认为appendonly.aof

RDB为全量持久化, AOF为增量持久化.

RDB的优势在于:

  1. 只包含一个文件, 方便备份.
  2. 方便转移, 直接将RDB文件移动到其他的存储介质上.
  3. 在数据量较多时, 使用RDB文件恢复速度比AOF快.
  4. RBD可以最大化Redis的性能, 父进程fork出一个子进程进行使用写时复制的方式进行保存工作, 父进程不执行任何持久化工作.

RDB的劣势在于:

  1. 保存一份RDB总是要间隔一段时间, 例如5分钟, 如果发生故障, 这段时间的数据会被丢失.
  2. 当数据集非常大的时候, fork过程可能非常耗时, 会在一段时间(n毫秒到1秒)内停止处理客户端请求.

AOF的优势在于:

  1. 更能保证耐久性, 因为每次需要追加的命令并不多, 因此可以频繁进行fsync(例如间隔1s), 这样在故障时丢失数据量更少.
  2. 如果AOF文件过大, Redis可以对其进行重写, 仅包含恢复当前数据集所需的最小命令集合.
  3. 有序地保存了所有写入操作, 且非常容易被人读懂, 因此更容易对其进行分析.

AOF的劣势在于:

  1. 对于相同数据集的备份, AOF文件的体积通常要大于RDB文件.
  2. 根据使用的fsync策略, Redis在使用AOF备份的速度可能慢于使用RDB的速度.

默认的备份方式是RDB方式.

Redis事务

使用MULTI指令开始一个事务, 之后所有命令被添加入事务队列, 直至使用EXEC触发事务或使用DISCARD取消事务.

WATCH指令可以被用于事务开始之前用以监测指定键, 如果在事务被触发前, 该键被更改, 则EXEC操作直接失败.

HyperLogLog

一个包含去重功能的计数器, 传入相同key不会增加计数, 但不太精确, 约有0.81%的误差.

HyperLogLog在数据超过某个阈值后, 占用的存储空间为12KB

布隆过滤器

Bloom Filter用以检索一个元素是否在某一个集合中,

值得注意的是, 布隆过滤器的数据结构中并不保存任何数据, 只是能够判断该数据是否被传入过.

因此, 实例化一个布隆过滤器, 在存储一条记录时之前先添加入布隆过滤器, 之后每次请求先检验该数据是否存在

如果存在, 在走缓存->数据库的查询流程, 如果不存在, 则直接返回.

布隆过滤器有一定误判几率, 某个不存在的记录可能被判定为存在, 但被布隆判为不存在的元素肯定不存在.

布隆过滤器所使用空间的大小与其误判率的要求相关, 当数据越来越多后, 为了维持误判率其占用空间也会上升.

布隆过滤器空间利用率很高, 当维持1%误判率时, 每个元素的判定所需空间约为9.6比特.

单线程

Redis的数据在内存中, 因此操作极快, 相较而言, 切换线程所耗费的时间就不容忽视, 并不能提高效率.

如果想利用多核, 可以启动多个Redis实例.

注: 这里的单线程指的是Redis进行数据操作使用单线程, 作为一个完整的服务, Redis本身包含多个线程.

非阻塞IO

在处理多个请求时, Redis单线程使用IO复用模型维持与多个客户端的链接.

Pipe

将多个独立的命令一次性发送, 然后一次性获得结果.

通过减少与Redis服务器的通信次数来实现降低往返时延, 最大程度利用服务端处理能力.

pipe的实现是在客户端, redis执行命令仍然是逐条进行的.

事务及其ACID特性

事务是数据库管理系统执行过程中的一个逻辑单位, 由一个有限的数据库操作序列组成.

ACID特性:

  • 原子性(Atomicity): 事务作为一个整体被执行, 包含在其中的对数据库的操作要么全部被执行, 要么都不执行.
  • 一致性(Consistency): 事务应确保数据库的状态从一个一致状态转换为另一个一致状态, 一致状态的含义是数据满足完整性约束.
  • 隔离性(Isolution): 多个事务并发执行时, 一个事务的执行不应影响其他事务的执行.
  • 持久性(Durability): 已被提交的事务对数据库的修改应当永远保存在数据库中.

缓存雪崩, 击穿, 穿透

Posted on 2018-09-21

缓存雪崩

Redis大量key同一时间大面积失效, 导致请求直接打到数据库上, 而导致数据库挂掉.

缓存雪崩的原因包括:

  1. 大量key在同一时间超时
  2. 缓存服务器宕机

解决方案

在批量向Redis存数据时, 将每个key的失效时间都添加一个随机数, 这样保证key不会在同一时间过期.

对热点数据, 不设置过期时间, 当内容变化时只进行更新操作.

使用集群缓存, 保证缓存服务的高可用. 在服务器宕掉之后, 对请求进行限流, 并尽快重建缓存.

缓存穿透

用户对数据库中不存在的数据发起频繁请求.

由于该数据不存在, 因此缓存中肯定不包含该数据, 这样仍然必须到数据库层次进行查询.

相当于用户请求直接穿透了缓存到达数据库, 如果频繁进行穿透, 数据库会很快挂掉

解决方案

  1. 缓存空值, 将请求的不存在的key设置为null并置于缓存中.
  2. 布隆过滤器, 首先判断该key的存在与否, 然后再决定是否进行缓存->数据库的查询.

以上两者可以结合进行.

缓存击穿

某个key被大规模请求, 每时每刻都承受着大量的压力.

当这个key失效的瞬间, 由于并发请求的数量极多, 因此该key还未来及重建的时候, 大量的请求就已经到达数据库.

解决方案

使用互斥锁

在从缓存中取数据失败时, 并不直接请求数据库, 而是设置一个与该key相关的lock(例如使用redis分布式锁).

所有请求必须先请求到这个锁, 然后才能访问数据库. 如果抢锁失败, 就重新请求缓存.

这样, 大量的请求只有一个能到达数据库, 等缓存重新建立起来之后, 这些请求重新从缓存中获得数据.

刷新key的超时时间

在key的value中保存一个值, 这个值指示着key的超时时间, 每次获得value的时候都检查这个值.

如果该值少于某个阈值, 立即刷新其超时时间, 这样保证热点数据能够一直存在于缓存.

缓存预热

在系统上线后先将一些热点数据直接放进缓存中, 而不是等到被请求时才放入缓存

编码问题

Posted on 2018-06-01

编码问题

什么是编码?从本质上说起

计算机元件上存储的是电位高低, 我们可以抽象地用1和0来表示高低电位, 这是最底层的东西.

通常我们称一位为bit, 代表一个电位的高低, 它只有两个状态, 高和低, 所以每一个bit可以区分两个不同信息.

因为, 每个字节, 即每8个位, 最多可以表示256种不同的含义, 这是最简单不过的数学题.

假如现在有一串电位 01000001 01000010已知这是一段文本信息, 那么代表的是什么含义?

没人知道, 也不可能有人知道.

唯一知道的就是, 在存储着这一串信息的内存区域, 其电位分布依次为高电平低电平低电平低电平 …..

ASCII编码

为了实现从电位序列到实际语义的转换, 我们定义了一组规范.

用7个连续电位代表一个字符(ASCII编码),显然, 7位能描述的是128种状态, 因此, ASCII字符最多有128个, 事实上也是如此.

鉴于英文字母只有26个, 其大小写状态再加上一些其他符号, 128个位置也是能应付得来的.

在建立了电位到二进制数字, 再到语义的映射, 即ASCII表之后, 我们可以理解了以上一串电位的含义了

01000001根据ASCII对照表, 代表字母A, 01000010则代表字母B, 所以这串电平代表的是两个字母’AB’

那么一个ASCII编码的字符在内存中占据7位吗?

不是的, 字节是计算机基本的存储单元, 一个字符最低也要占据1个字节(8个bit), 所以其首位固定添加一个0以占据一个字节.

ASCII编码的局限性

当计算机的发展不限于英语国家乃至西欧国家时, 128个字符位就不够用了

尤其对于非字母式语言, 可能需要上万个位置来存放自己的字符, 各种语言加起来, 这个数量就更多了

如果计算机仍然只使用ASCII编码的话, 那么除了英语其他语言都无法在计算机上显示

为了解决这个问题, 各国都推出了解决方案, 解决了部分问题, 也在互相影响下引入了歧义, 带来了混乱.

我们直接跳到最终的解决方案 unicode编码

unicode

unicode是一个编码规范.

在设想中, unicode映射表极为庞大, 庞大到所有语言的字符乃至今后世界上可能出现的所有字符加起来都不足以占满它的位置.

我们只需要指定一个序号, 然后根据映射表, 就确定表示的究竟是哪个字符.

没有争抢, 没有歧义, 不需要任何转换技巧, 所有语言的字符一视同仁.

需要说明的是:unicode是一个编码规范, 它定义了映射表, 但并不是实际的编码方案, 它的实际实现是UTF-8以及utf-16以及utf-32

UTF-8、UTF-16以及UTF-32

第一个版本unicode 1.0(1994年)中打算用两个字节表示世界上所有的字符(也就是2的16次幂一共65536个空位, 显然这是一个错误的估计)

utf-16是unicode1.0最自然的编码实现, 使用16位来指定一个字符, 无论这个字符的序号是多么地小, 它都占据两个字节, 而且在内存和存储中均保持一致

显然这种方法存在缺陷, 对于序号较小的字符(例如字母A,序号为65即1000001), 其占用的两个字节的第一个字节中全是0, 且无法兼容ASCII.

别忘了, 此时网络上绝大部分文本都是英文, 所有英文字母占用的空间翻倍, 会带来传输和存储过程效率减半的严重后果.

而另一种实现utf-8就是为了解决这个问题：

在内存中时, 仍然使用两个字节来表示一个字符, 但在转换为字节流时, utf-8使用最少8位来代表一个字符

也就是说, 一个字符至少映射为1个字节, 可能是多个字节, 只需要根据其开头序列, 即可知道应将几个字节视为一个字符.

那么解码的时候, 怎么确定当前的字节代表一个字符, 还是应当与之后的字节连在一起代表另一个字符呢?

答案也很简单, 抛弃一些空位, 以避免出现任何歧义.

unicode编码(十六进制) utf-8字节流(二进制)
000000-00007F 0xxxxxxx
000080-0007FF 110xxxxx 10xxxxxx
000800-00FFFF 1110xxxx 10xxxxxx 10xxxxxx
010000-10FFFF 11110xxx 10xxxxxx 10xxxxxx 10xxxxxx
… …

我们可以看到, 在多个字节表示一个字符时, 一些空位是无法被使用的.

unicode2.0则是使用4个字节来表示一个字符的编码形式, 可用编码空间有二十多个亿个.

实际中使用的unicode字符集的序号则不超过0x10FFFF,

如同utf-16是unicode1.0的自然实现, utf-32是unidoce2.0的自然实现.

当然, 这种编码方式很少被使用, 大家都用utf-8这种可变长的编码方式.

编码过程与解码过程

编码:将字符转换为字节. 从存储或字节流中将字节转换为字符, 就要进行解码

解码:将字节转换为字符. 从内存中将字符转换为字节流用于存储或传输, 就要进行编码

BOM是什么

(Byte Order Mark)字节序

LITTLE-ENDIAN（小字节序、低字节序）,即低位字节排放在内存的低地址端, 高位字节排放在内存的高地址端. 与之对应的是：BIG-ENDIAN(大字节序、高字节序)

utf-8以字节为编码单元, 无需使用这个来确定字节序, 而utf-16中以2个字节为编码单元, 因此有必要先确定字节序.

/boot 开机所需文件，内核、开机菜单及所需配置文件等
/dev 以文件形式存放的设备与接口
/etc 配置文件，所有的配置文件都应当放在这个目录
/bin 二进制可执行文件，用户指令，常用命令一般都在此处
/sbin 开机过程中需要的命令，只有超级用户能访问的二进制可执行文件
/usr Unix System Resource 用于存放系统应用程序
/srv 一些网络服务的数据文件目录
/var 运行时会改变的文件，例如日志等
/root 超级用户的主目录
/home 存放用户的根目录，用户主目录的基点
/media 用于挂载设备文件的目录
/opt 额外可选的应用程序包
/temp 临时文件的存储目录
/mnt 系统管理员安装临时文件系统的安装点

另外, /usr目录中各子目录和应当放置的文件内容

/usr子目录 应当放置的文件内容
/bin 众多的应用程序
/lib 常用的动态链接库和软件包的配置文件
/src 源代码
/local 用户自行安装的软件
/local/bin 本地增加的命令
/local/lib 本地增加的库

Linux文件权限修改

Posted on 2018-05-21

文件权限与查看

使用ls查看文件信息, 常用参数为

  • a 查看所有文件
  • l 查看详细信息
  • d 查看目录信息

文件包括以下类型:

  • - 普通文件
  • d 目录文件
  • l 链接文件
  • b 块设备文件
  • c 字符设备文件
  • p 管道文件

每个文件都有所属的用户和组, 文件的基本权限有三个:

  • r 读
  • w 写
  • x 执行

对于一个目录文件来说, 这三项的含义为:

  • r: 可以读取目录内文件列表
  • w: 可以在该目录增删改文件
  • x: 可以进入该目录

权限与数字的对应关系为

  • r – 4
  • w – 2
  • x – 1

所以一个rwx权限可以等价表示为4+2+1=7, 依次类推，可读可写为4+2=6，只读为4

在ls命名列出的文件列表中, 权限从左到右分为三组, 分别是

  1. 文件所有者
  2. 文件所属组
  3. 其他用户

对该文件具有的权限.

修改文件权限指令

chmod [选项] 权限 文件

例如将某个文件修改为所有用户可读可写可执行

用户和用户组

每个用户在创建时都会自动创建一个同名的基本用户组, 该基本用户组只包含该用户.

如果用户以后纳入其他用户组, 则对于该用户来说属于扩展用户组.

一个用户只能有一个基本用户组, 但可以有多个扩展用户组.
