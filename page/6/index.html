RESTful风格

Posted on 2019-02-17

REST标准

REST是REpresentational State Transfer的缩写, 意味着资源通过表现层进行状态的转移.

符合REST标准的架构, 就可以称为RESTful风格的web服务.

  1. 资源: 网络上的一个实体, 每种资源对应一个特定的URI.
  2. 表现层: 对资源的描述, 可以用json, xml等形式.
  3. 状态转移: 对资源当前的状态进行更改, 例如删除, 创建, 更新等.

即客户端通过不同的Http method传递json/xml等格式的数据来告知服务端应当如何操作资源.

RESTful 设计指南

协议(protocal)

总是使用HTTPS协议

域名(domain)

尽量部署在专用域名下, 例如api.example.com

版本(version)

应该包含版本号, 放在URL或HTTP头中, 例如api.example.com/v1

路径(endpoint)

指向某种资源的具体路径. 注意路径中不包含动词, 因为URI只代表资源本身, 例如:

  • /api/v1/products/ 所有产品
  • /api/v1/products/123 主键为123的产品

HTTP动词(verbs)

合理使用不同的HTTP METHOD来表示对资源的操作

  • GET(retrieve or list): 取回资源, 一个或多个
  • POST(create): 创建资源
  • PUT(update): 更新现有资源, 客户端必须提供完整的资源描述.
  • PATCH(partical update) 部分更新现有资源, 客户端只需提供部分资源描述
  • DELETE(destory): 删除现有资源

以及另外两个特殊的METHOD:

  • HEAD: 获取响应的元数据
  • OPTION: 获取资源的信息, 包括资源路径, 所支持的操作等等.

过滤(filtering)

客户端可以通过指定过滤条件让服务器返回自己所需的部分资源, 例如:

  • ?limit=10 限制返回资源的数量
  • ?offset=10 从第10项开始返回资源
  • ?page=2&page_size=100 指定每页包含的数据量以及目标页
  • ?ordering=name 指定资源排序依据
  • ?type=ball 指定资源类别

状态码(status)

使用准确的状态码来表示服务端动作, 具体根据http标准.

错误处理(error handling)

当请求出错, 服务端应当包含足够的信息指出错误之处及出错原因.

超链接(hyperlink)

最好在响应中包含相关资源的超链接而非标识符, 使客户端能够直接请求相关资源的endpoint.

数据格式

最好使用JSON, 避免使用XML.

授权

授权应当使用OAuth2.0框架.

REST级别

0级: Swamp of POX 传统的RPC, 基于SOAP的WS调用的服务名, 参数放在HTTP协议的body里面, 同时必须以POST方式提交, 问题在于你必须清楚的知道所有服务, 子服务, 及其参数的信息, 并且需要知道各种服务的不同点.

1级: Resources 利用resource概念, 把所有服务都抽取成resource概念, 从body中提取到header里, 这样做的好处就是如果你知道一个服务地址, 你可能无需知道具体服务是什么, 依照资源的惯例就访问到服务, 比如/books/1.

2级: HTTP verbs 利用HTTP动词, HTTP定义了4种动词, GET获取服务器资源, POST在服务器上创建新资源, PUT更改服务器上资源, DELETE删除服务器上资源, 任何操作都可以看成增删改查, 所以利用标准的http verb加上resource（/book/1）就能准确地操作资源, 当你不知道服务具体是什么的时候也可以轻易按照惯例访问到服务, 然而服务供应商更改服务也需要遵循惯例, 不会像RPC那样轻易更改服务接口.

3级: Hypermedia controls 最高级别, 超媒体即应用状态引擎. 这个意思是说, 对于任何服务都存在很多子服务, 你只需要知道第一个服务的入口, 便可以依据服务返回结构的自描述性得到下一个服务的入口, 这样在服务供应商修改服务的时候, 不会影响到客户端的调用.

127.0.0.1和0.0.0.0

Posted on 2019-02-01

127.0.0.1 是环回地址, 通常分配给loopback处理, 对127.0.0.1的访问都会指向本机, 所以通常用于本机应用之间的交互,

对127.0.0.1的请求不会出入网卡, 网络设备也不会对其路由.

操作系统初始化本机的TCP/IP协议栈时, 设置协议栈本身的IP地址为127.0.0.1, 并注入路由表, 当IP层接到目的地址为该地址的的数据包时, 直接转发到本机的IP层进行处理.

即, 对127.0.0.1的请求所使用的最底层协议栈为TCP/IP层, 直接在网络层和传输层之间环回, 不会通过链路层和物理层.

能ping通127.0.0.1意味着TCP/IP协议栈运行正常.

localhost通常被设置为127.0.0.1的域名, 在支持ipv6的设备中, 同时还指向[::1]

使用localhost与127.0.0.1没有任何区别.

0.0.0.0指的是任意ip地址IP_ADDR_ANY, 也包括环回地址, 主机监听0.0.0.0时, 会接受目标地址为任意地址的请求.

0.0.0.0永远不能ping通.

在Docker容器中, 127.0.0.1指的是本容器的网络地址, 监听127.0.0.1的容器内服务, 不会接收任何非本容器内的请求, 包括其他容器的请求也不会被接收.

Redis Cluster 集群

Posted on 2019-01-27

Redis Cluster

Redis Sentinel 着眼于高可用, 但Redis实例的容量仍然受限于单机的内存容量.

Cluster提供将多个redis实例作为一个整体, 使用哈希槽的概念, 将键分配在多个redis实例中, 这样redis的容量就不再受限于单个机器的内存.

即, Cluster解决的是Redis的性能和扩展性问题.

Redis集群中的每个redis实例称为一个节点, 在建立集群后, 为每个节点设置主从复制, 并在合适的时机进行主从切换以提高可用性.

Cluster的主从切换并不需要使用Sentinel, 而是通过多个节点之间的通信协商.

使用集群的流程为

  1. 创建多个Redis节点
  2. 创建集群
  3. 客户端连接

创建多个节点

每个节点是一个redis实例, 官方给出作为一个节点的最小配置项如下:

cluster-config-file 指定保存节点信息状态的文件, 我们只需要为这个文件指名, 但不需要手动管理该文件, 会由Cluster进行创建和随时更新.
cluster-node-timeout 指定在一个节点在无响应多少毫秒后视为不可达.

按照推荐, 我们使用docker启动多个实例

启动6个实例, 端口号分布在7000到7005之间.

每个启动的实例会分配一个id, 用作该实例的唯一标识, 每个节点靠ID来记录每个其他节点, 而不是IP和端口, 因为IP和端口可能会变化, 但唯一的节点标识符在整个节点的生命周期中保持不变, 这个ID称之为节点ID(Node ID).

访问集群

访问集群的操作与访问单个redis实例不同, 因为每个节点都只负责一部分槽.

可以访问任何一个节点, 但除非所读写的键正好在该节点所分配的槽的范围内, 否则节点会返回一个Moved错误, 并给出该槽被分配在节点的地址.

因此, 需要一个封装良好的客户端, 来进行对集群的操作, Python程序中可以使用redis-py-cluster包.

或是使用redis-cli进行读写测试, 注意启动时增加了-c参数以连接cluster.

TCP连接

每个节点会打开两个TCP连接, 一个进行数据操作的端口, 另一个10000偏移量的端口则进行节点间的通信.

包括检测错误, 更新配置, 故障转移授权等, 客户端不应该尝试连接第二个端口, 但要保证该端口开放.

数据分片

Redis不使用一致性hash, 而是使用哈希槽(hash slot)的概念, 每个集群共拥有16384个槽, 对键进行CRC16编码然后对16384取模即可获得该键所属槽位.

集群中的每个节点负责哈希槽的一个子集, 例如在一个3个主节点的集群中, 每个节点负责的槽范围为:

  • 节点A: 0-5500
  • 节点B: 5501-11000
  • 节点C: 11001-16384

使用槽的概念让添加和移除节点非常容易, 例如增加一个节点D, 只需要从A, B, C中移动一些槽到节点D即可.

而如果要删除A节点, 则将其所有的槽到B和C. 当A节点被清空之后, 就可以将其删除.

移动槽的过程不需要停止操作, 所以添加移除节点, 或是更改某个节点哈希槽的百分比, 不需要任何停机时间.

键hash标签

Hash Tag指的是一个用于计算键哈希槽的部分的标记, 默认是{}, 即键中包含{}时, 仅{}内的部分会用于计算槽位

Redis Cluster并不支持涉及多个节点的多键操作, 因此采用Hash Tag可以保证采用了相同Hash Tag的多个键被分配至同一个槽位.

当然, 如果因为键中本来就需要包含不作为键hash标志的{}, 根据规则也可以绕开这个机制, 详情查看官方文档

客户端的数据访问

分片之后, 盲目对节点进行读写, 大概率不会恰好命中键哈希槽所属的节点, 这意味着我们需要与redis服务器通信两次

  1. 发起请求读写请求, 失败, 但获得了键的槽位和该槽所属的节点的地址
  2. 再次发起请求, 读写成功.

实际上, 为了避免这种情况, Cluster配置了slot表, 客户端首先通过slots命令获得slot表, 获知各个槽位由哪些节点负责, 之后的请求就可以一发中的.

当然, 槽位不是一成不变的, 客户端需要在合适的时机(例如设置一个超时时间, 或是在多次未直接命中后)更新这个表.

集群的主从模型

为集群的每个节点建立从节点, 保证了每个节点的高可用.

Redis集群的主从切换并不由外置的Sentinel作为监控, 而是由集群中每个节点共同执行.

自动主从切换的前提是半数以上的master存活, 这就是为什么官方推荐至少要3个以上的节点作为一个集群.

集群的一致性保证

Redis集群并不保证强一致性, 以下情况可能发生数据不一致, 例如:

  1. 客户端写入主节点
  2. 主从复制尚未完成时主节点挂掉
  3. 从节点被提升为主节点

此时这个写操作就永久地丢失了. 又或者:

  1. 发生了网络分割, 少数主节点被孤立,
  2. 在节点超时(node timeout)之前, 客户端继续写入这些主节点
  3. 多数主节点注意到少数主节点不可达, 于是将其附属的从节点提升为主节点

这样, 在整个集群未确认少数主节点不可达的这段时间内, 客户端对这些异常主节点的写操作就丢失了.

不过情况并不会太糟, 在节点超时之后, 被分割的少数主节点仍然无法感知到集群中多数主节点, 会自动停止接受写请求.
