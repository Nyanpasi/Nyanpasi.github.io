RESTful风格

Posted on 2019-02-17

REST标准

REST是REpresentational State Transfer的缩写, 意味着资源通过表现层进行状态的转移.

符合REST标准的架构, 就可以称为RESTful风格的web服务.

  1. 资源: 网络上的一个实体, 每种资源对应一个特定的URI.
  2. 表现层: 对资源的描述, 可以用json, xml等形式.
  3. 状态转移: 对资源当前的状态进行更改, 例如删除, 创建, 更新等.

即客户端通过不同的Http method传递json/xml等格式的数据来告知服务端应当如何操作资源.

RESTful 设计指南

协议(protocal)

总是使用HTTPS协议

域名(domain)

尽量部署在专用域名下, 例如api.example.com

版本(version)

应该包含版本号, 放在URL或HTTP头中, 例如api.example.com/v1

路径(endpoint)

指向某种资源的具体路径. 注意路径中不包含动词, 因为URI只代表资源本身, 例如:

  • /api/v1/products/ 所有产品
  • /api/v1/products/123 主键为123的产品

HTTP动词(verbs)

合理使用不同的HTTP METHOD来表示对资源的操作

  • GET(retrieve or list): 取回资源, 一个或多个
  • POST(create): 创建资源
  • PUT(update): 更新现有资源, 客户端必须提供完整的资源描述.
  • PATCH(partical update) 部分更新现有资源, 客户端只需提供部分资源描述
  • DELETE(destory): 删除现有资源

以及另外两个特殊的METHOD:

  • HEAD: 获取响应的元数据
  • OPTION: 获取资源的信息, 包括资源路径, 所支持的操作等等.

过滤(filtering)

客户端可以通过指定过滤条件让服务器返回自己所需的部分资源, 例如:

  • ?limit=10 限制返回资源的数量
  • ?offset=10 从第10项开始返回资源
  • ?page=2&page_size=100 指定每页包含的数据量以及目标页
  • ?ordering=name 指定资源排序依据
  • ?type=ball 指定资源类别

状态码(status)

使用准确的状态码来表示服务端动作, 具体根据http标准.

错误处理(error handling)

当请求出错, 服务端应当包含足够的信息指出错误之处及出错原因.

超链接(hyperlink)

最好在响应中包含相关资源的超链接而非标识符, 使客户端能够直接请求相关资源的endpoint.

数据格式

最好使用JSON, 避免使用XML.

授权

授权应当使用OAuth2.0框架.

REST级别

0级: Swamp of POX 传统的RPC, 基于SOAP的WS调用的服务名, 参数放在HTTP协议的body里面, 同时必须以POST方式提交, 问题在于你必须清楚的知道所有服务, 子服务, 及其参数的信息, 并且需要知道各种服务的不同点.

1级: Resources 利用resource概念, 把所有服务都抽取成resource概念, 从body中提取到header里, 这样做的好处就是如果你知道一个服务地址, 你可能无需知道具体服务是什么, 依照资源的惯例就访问到服务, 比如/books/1.

2级: HTTP verbs 利用HTTP动词, HTTP定义了4种动词, GET获取服务器资源, POST在服务器上创建新资源, PUT更改服务器上资源, DELETE删除服务器上资源, 任何操作都可以看成增删改查, 所以利用标准的http verb加上resource（/book/1）就能准确地操作资源, 当你不知道服务具体是什么的时候也可以轻易按照惯例访问到服务, 然而服务供应商更改服务也需要遵循惯例, 不会像RPC那样轻易更改服务接口.

3级: Hypermedia controls 最高级别, 超媒体即应用状态引擎. 这个意思是说, 对于任何服务都存在很多子服务, 你只需要知道第一个服务的入口, 便可以依据服务返回结构的自描述性得到下一个服务的入口, 这样在服务供应商修改服务的时候, 不会影响到客户端的调用.

127.0.0.1和0.0.0.0

Posted on 2019-02-01

127.0.0.1 是环回地址, 通常分配给loopback处理, 对127.0.0.1的访问都会指向本机, 所以通常用于本机应用之间的交互,

对127.0.0.1的请求不会出入网卡, 网络设备也不会对其路由.

操作系统初始化本机的TCP/IP协议栈时, 设置协议栈本身的IP地址为127.0.0.1, 并注入路由表, 当IP层接到目的地址为该地址的的数据包时, 直接转发到本机的IP层进行处理.

即, 对127.0.0.1的请求所使用的最底层协议栈为TCP/IP层, 直接在网络层和传输层之间环回, 不会通过链路层和物理层.

能ping通127.0.0.1意味着TCP/IP协议栈运行正常.

localhost通常被设置为127.0.0.1的域名, 在支持ipv6的设备中, 同时还指向[::1]

使用localhost与127.0.0.1没有任何区别.

0.0.0.0指的是任意ip地址IP_ADDR_ANY, 也包括环回地址, 主机监听0.0.0.0时, 会接受目标地址为任意地址的请求.

0.0.0.0永远不能ping通.

在Docker容器中, 127.0.0.1指的是本容器的网络地址, 监听127.0.0.1的容器内服务, 不会接收任何非本容器内的请求, 包括其他容器的请求也不会被接收.

Redis基础

Posted on 2019-01-25

基本数据类型

Redis的键为字符串, 而值有5种基本数据类型:

  1. 字符串(String)
  2. 列表(List)
  3. 字典(Hash)
  4. 集合(Set)
  5. 有序集合(Sorted Set)

字符串

Redis字符串为动态字符串, 采用预分配冗余空间的方式来减少内存的频繁分配.

当字符串长度小于1M时, 扩容都是加倍现有的空间, 如果超过1M, 扩容时一次只会多扩1M的空间.

需要注意的是字符串最大长度为512M.

另外, 字符串的索引从0开始。

列表(List)

Redis的列表为双向链表, 首尾插入和删除较快, 而定位操作慢.

数据较少时, 使用连续内存存储, 这个结构是ziplist, 数据较多时, 使用多个链接起来的ziplist组成的quicklist.

使用列表可以方便地实现队列或者栈.

字典(hash)

一个无序字典, 字典的值只允许是字符串. 解决键冲突的方式为拉链法.

为了让负载因子维持在一个合理的范围内, 在数据增加或减少到一定程度后对其进行响应的扩展或收缩.

渐进式rehash

扩展或收缩哈希表需要将旧的hashtable里面的所有键值对rehash到新的hashtable里面

如果键值对数量很多时, 一次性rehash引起的庞大的计算量导致服务器在一段时间内停止服务.

因此, rehash动作并不是一次性、集中式地完成, 而是分多次、渐进式地完成的.

集合(set)

一个值为null的字典.

有序集合(zset)

每个元素都关联一个double类型的分数, redis通过分数进行集合成员的排序

键与分数之间的关系是用字典来实现的, 即分数作为元素的值, 但因为这个值只能是一个double类型的分数, 所以我们称之为set, 而非dict.

查询某个元素的score, 直接通过dict查询, 时间复杂度为O(1)

跳跃列表(skiplist)

zset内部排序功能是通过跳跃列表实现,

跳跃列表本质上是一种查找结构, 用于解决查找问题.

具体的实现是一个多层链表, 分为不同的层级, 相同层级的元素都使用指针进行连接.

因此某些节点同时归属于不同层级, 该节点也就包含多个后向指针.

每个节点至少包含一个前向指针, 并包含至少一个(可能多个)后向指针, 具体取决于该节点究竟同时在多少层.

  1. 最底层即L0层, 包含所有的元素
  2. 次底层即L1层, 每个元素有50%的机会进入这一层
  3. 第三层即L2层, 每个元素有25%的机会进入这一层
  4. 以此类推, 最多有32层.

显然, 最底层是一个双向链表, 而上层则不然.

每次的搜索过程从现存的最高层开始, 向下逐级搜索.

由于所有节点的晋升依赖于概率, 无需维持各层之间的数量约束关系, 因此插入效率比平衡树类型的索引要高.

而查找的平均时间复杂度是O(logn), 与平衡树类型相当.

查询元素名次, 通过分数反查元素范围等, 因为要使用skiplist, 所以时间复杂度为O(logn+M), M是要返回的元素的个数.

Redis淘汰策略

六种淘汰机制, 使用maxmemory-policy设置项进行设置

maxmemory-policy 执行策略
noeviction 不进行淘汰
allkeys-lru 对所有key按照LRU原则进行淘汰
allkeys-random 对所有key进行随机淘汰
volatile-lru 对易失key按照LRU原则进行淘汰
volatile-random 对易失key进行随机淘汰
volatile-ttl 对易失key按照生存时间最少进行淘汰

注:

  1. LRU(Least Recently Used) 最近最少使用
  2. TTL(Time To Live) 生存时间
  3. volatile: 设置了超时的key被视为volatile, 即易失的

Redis 持久化机制

  1. RDB机制, 指定时间间隔内将内存中的快照集快照写入磁盘. 默认为dump.rdb
  2. AOF机制, Append Only File, 每个写命令都通过write追加到文件中. 默认为appendonly.aof

RDB的优势在于:

  1. 只包含一个文件, 方便备份.
  2. 方便转移, 直接将RDB文件移动到其他的存储介质上.
  3. 在数据量较多时, 使用RDB文件恢复速度比AOF快.
  4. RBD可以最大化Redis的性能, 父进程fork出一个子进程进行保存工作, 父进程无需执行任何IO.

RDB的劣势在于:

  1. 保存一份RDB总是要间隔一段时间, 例如5分钟, 如果发生故障, 这段时间的数据会被丢失.
  2. 当数据集非常大的时候, fork过程可能非常耗时, 会在一段时间(n毫秒到1秒)内停止处理客户端请求.

AOF的优势在于:

  1. 更能保证耐久性, 因为每次需要追加的命令并不多, 因此可以频繁进行fsync(例如间隔1s), 这样在故障时丢失数据量更少.
  2. 如果AOF文件过大, Redis可以对其进行重写, 仅包含恢复当前数据集所需的最小命令集合.
  3. 有序地保存了所有写入操作, 且非常容易被人读懂, 因此更容易对其进行分析.

AOF的劣势在于:

  1. 对于相同数据集的备份, AOF文件的体积通常要大于RDB文件.
  2. 根据使用的fsync策略, Redis在使用AOF备份的速度可能慢于使用RDB的速度.

默认的备份方式是RDB方式.

Redis事务

使用MULTI指令开始一个事务, 之后所有命令被添加入事务队列, 直至使用EXEC触发事务或使用DISCARD取消事务.

WATCH指令可以被用于事务开始之前用以监测指定键, 如果在事务被触发前, 该键被更改, 则EXEC操作直接失败.

HyperLogLog

一个包含去重功能的计数器, 去重依据是设置值时的参数.

布隆过滤器

用以检索一个元素是否在某一个集合中,

值得注意的是, 布隆过滤器的数据结构中并不保存任何数据, 只是能够判断该数据是否被传入过.

因此, 实例化一个布隆过滤器, 在存储一条记录时之前先添加入布隆过滤器, 之后每次请求先检验该数据是否存在

如果存在, 在走缓存->数据库的查询流程, 如果不存在, 则直接返回.

布隆过滤器有一定误判几率, 某个不存在的记录可能被判定为存在, 但被布隆判为不存在的元素肯定不存在.

布隆过滤器所使用空间的大小与其误判率的要求相关, 当数据越来越多后, 为了维持误判率其占用空间也会上升.

布隆过滤器空间利用率很高, 当维持1%误判率时, 每个元素的判定所需空间约为9.6比特.

Redis实现分布式锁

实现思路: 使用setnx(set if not exists)指令来设置某个键, 如果成功, 则视为抢到了锁.

在执行完代码后, 使用del指令释放锁.

为了确保在程序出错时, 锁一定被释放, 还应该为其设置超时时间, 并且, 抢锁和设置超时需要为原子操作.

Redis为什么设计为单线程?

Redis的数据在内存中, 因此操作极快, 相较而言, 切换线程所耗费的时间就不容忽视, 并不能提高效率.

因此, 只有在有必要进行多任务而不是增加效率的情况下, 才会启动多个线程.

如果想利用多核, 可以启动多个Redis实例.

注: 这里的单线程指的是Redis进行数据操作的是单线程, 作为一个完整的服务, Redis本身包含多个线程.

事务及其ACID特性

事务是数据库管理系统执行过程中的一个逻辑单位, 由一个有限的数据库操作序列组成.

ACID特性:

  • 原子性(Atomicity): 事务作为一个整体被执行, 包含在其中的对数据库的操作要么全部被执行, 要么都不执行.
  • 一致性(Consistency): 事务应确保数据库的状态从一个一致状态转换为另一个一致状态, 一致状态的含义是数据满足完整性约束.
  • 隔离性(Isolution): 多个事务并发执行时, 一个事务的执行不应影响其他事务的执行.
  • 持久性(Durability): 已被提交的事务对数据库的修改应当永远保存在数据库中.

缓存雪崩, 击穿, 穿透

Posted on 2018-09-21

缓存雪崩

Redis大量key同一时间大面积失效, 导致请求直接打到数据库上, 而导致数据库挂掉.

缓存雪崩的原因包括:

  1. 大量key在同一时间超时
  2. 缓存服务器宕机

解决方案

在批量向Redis存数据时, 将每个key的失效时间都添加一个随机数, 这样保证key不会在同一时间过期.

对热点数据, 不设置过期时间, 当内容变化时只进行更新操作.

使用集群缓存, 保证缓存服务的高可用. 在服务器宕掉之后, 对请求进行限流, 并尽快重建缓存.

缓存穿透

用户对数据库中不存在的数据发起频繁请求.

由于该数据不存在, 因此缓存中肯定不包含该数据, 这样仍然必须到数据库层次进行查询.

相当于用户请求直接穿透了缓存到达数据库, 如果频繁进行穿透, 数据库会很快挂掉

解决方案

  1. 缓存空值, 将请求的不存在的key设置为null并置于缓存中.
  2. 布隆过滤器, 首先判断该key的存在与否, 然后再决定是否进行缓存->数据库的查询.

以上两者可以结合进行.

缓存击穿

某个key被大规模请求, 每时每刻都承受着大量的压力.

当这个key失效的瞬间, 由于并发请求的数量极多, 因此该key还未来及重建的时候, 大量的请求就已经到达数据库.

解决方案

使用互斥锁

在从缓存中取数据失败时, 并不直接请求数据库, 而是设置一个与该key相关的lock(例如使用redis分布式锁).

所有请求必须先请求到这个锁, 然后才能访问数据库. 如果抢锁失败, 就重新请求缓存.

这样, 大量的请求只有一个能到达数据库, 等缓存重新建立起来之后, 这些请求重新从缓存中获得数据.

刷新key的超时时间

在key的value中保存一个值, 这个值指示着key的超时时间, 每次获得value的时候都检查这个值.

如果该值少于某个阈值, 立即刷新其超时时间, 这样保证热点数据能够一直存在于缓存.

缓存预热

在系统上线后先将一些热点数据直接放进缓存中, 而不是等到被请求时才放入缓存
