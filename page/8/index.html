Posted on 2018-06-01

编码问题

什么是编码?从本质上说起

计算机元件上存储的是电位高低, 我们可以抽象地用1和0来表示高低电位, 这是最底层的东西.

通常我们称一位为bit, 代表一个电位的高低, 它只有两个状态, 高和低, 所以每一个bit可以区分两个不同信息.

因为, 每个字节, 即每8个位, 最多可以表示256种不同的含义, 这是最简单不过的数学题.

假如现在有一串电位 01000001 01000010已知这是一段文本信息, 那么代表的是什么含义?

没人知道, 也不可能有人知道.

唯一知道的就是, 在存储着这一串信息的内存区域, 其电位分布依次为高电平低电平低电平低电平 …..

ASCII编码

为了实现从电位序列到实际语义的转换, 我们定义了一组规范.

用7个连续电位代表一个字符(ASCII编码),显然, 7位能描述的是128种状态, 因此, ASCII字符最多有128个, 事实上也是如此.

鉴于英文字母只有26个, 其大小写状态再加上一些其他符号, 128个位置也是能应付得来的.

在建立了电位到二进制数字, 再到语义的映射, 即ASCII表之后, 我们可以理解了以上一串电位的含义了

01000001根据ASCII对照表, 代表字母A, 01000010则代表字母B, 所以这串电平代表的是两个字母’AB’

那么一个ASCII编码的字符在内存中占据7位吗?

不是的, 字节是计算机基本的存储单元, 一个字符最低也要占据1个字节(8个bit), 所以其首位固定添加一个0以占据一个字节.

ASCII编码的局限性

当计算机的发展不限于英语国家乃至西欧国家时, 128个字符位就不够用了

尤其对于非字母式语言, 可能需要上万个位置来存放自己的字符, 各种语言加起来, 这个数量就更多了

如果计算机仍然只使用ASCII编码的话, 那么除了英语其他语言都无法在计算机上显示

为了解决这个问题, 各国都推出了解决方案, 解决了部分问题, 也在互相影响下引入了歧义, 带来了混乱.

我们直接跳到最终的解决方案 unicode编码

unicode

unicode是一个编码规范.

在设想中, unicode映射表极为庞大, 庞大到所有语言的字符乃至今后世界上可能出现的所有字符加起来都不足以占满它的位置.

我们只需要指定一个序号, 然后根据映射表, 就确定表示的究竟是哪个字符.

没有争抢, 没有歧义, 不需要任何转换技巧, 所有语言的字符一视同仁.

需要说明的是:unicode是一个编码规范, 它定义了映射表, 但并不是实际的编码方案, 它的实际实现是UTF-8以及utf-16以及utf-32

UTF-8、UTF-16以及UTF-32

第一个版本unicode 1.0(1994年)中打算用两个字节表示世界上所有的字符(也就是2的16次幂一共65536个空位, 显然这是一个错误的估计)

utf-16是unicode1.0最自然的编码实现, 使用16位来指定一个字符, 无论这个字符的序号是多么地小, 它都占据两个字节, 而且在内存和存储中均保持一致

显然这种方法存在缺陷, 对于序号较小的字符(例如字母A,序号为65即1000001), 其占用的两个字节的第一个字节中全是0, 且无法兼容ASCII.

别忘了, 此时网络上绝大部分文本都是英文, 所有英文字母占用的空间翻倍, 会带来传输和存储过程效率减半的严重后果.

而另一种实现utf-8就是为了解决这个问题：

在内存中时, 仍然使用两个字节来表示一个字符, 但在转换为字节流时, utf-8使用最少8位来代表一个字符

也就是说, 一个字符至少映射为1个字节, 可能是多个字节, 只需要根据其开头序列, 即可知道应将几个字节视为一个字符.

那么解码的时候, 怎么确定当前的字节代表一个字符, 还是应当与之后的字节连在一起代表另一个字符呢?

答案也很简单, 抛弃一些空位, 以避免出现任何歧义.

unicode编码(十六进制) utf-8字节流(二进制)
000000-00007F 0xxxxxxx
000080-0007FF 110xxxxx 10xxxxxx
000800-00FFFF 1110xxxx 10xxxxxx 10xxxxxx
010000-10FFFF 11110xxx 10xxxxxx 10xxxxxx 10xxxxxx
… …

我们可以看到, 在多个字节表示一个字符时, 一些空位是无法被使用的.

unicode2.0则是使用4个字节来表示一个字符的编码形式, 可用编码空间有二十多个亿个.

实际中使用的unicode字符集的序号则不超过0x10FFFF,

如同utf-16是unicode1.0的自然实现, utf-32是unidoce2.0的自然实现.

当然, 这种编码方式很少被使用, 大家都用utf-8这种可变长的编码方式.

编码过程与解码过程

编码:将字符转换为字节. 从存储或字节流中将字节转换为字符, 就要进行解码

解码:将字节转换为字符. 从内存中将字符转换为字节流用于存储或传输, 就要进行编码

BOM是什么

(Byte Order Mark)字节序

LITTLE-ENDIAN（小字节序、低字节序）,即低位字节排放在内存的低地址端, 高位字节排放在内存的高地址端. 与之对应的是：BIG-ENDIAN(大字节序、高字节序)

utf-8以字节为编码单元, 无需使用这个来确定字节序, 而utf-16中以2个字节为编码单元, 因此有必要先确定字节序.

Linux目录内容

Posted on 2018-05-21

根据FHS(Filesystem Hierarchy Standard, 文件系统层次化标准)，Linux系统的目录及其内容应当如下

目录 应当放置的文件内容
/boot 开机所需文件，内核、开机菜单及所需配置文件等
/dev 以文件形式存放的设备与接口
/etc 配置文件，所有的配置文件都应当放在这个目录
/bin 二进制可执行文件，用户指令，常用命令一般都在此处
/sbin 开机过程中需要的命令，只有超级用户能访问的二进制可执行文件
/usr Unix System Resource 用于存放系统应用程序
/srv 一些网络服务的数据文件目录
/var 运行时会改变的文件，例如日志等
/root 超级用户的主目录
/home 存放用户的根目录，用户主目录的基点
/media 用于挂载设备文件的目录
/opt 额外可选的应用程序包
/temp 临时文件的存储目录
/mnt 系统管理员安装临时文件系统的安装点

另外, /usr目录中各子目录和应当放置的文件内容

/usr子目录 应当放置的文件内容
/bin 众多的应用程序
/lib 常用的动态链接库和软件包的配置文件
/src 源代码
/local 用户自行安装的软件
/local/bin 本地增加的命令
/local/lib 本地增加的库

Linux文件权限修改

Posted on 2018-05-21

文件权限与查看

使用ls查看文件信息, 常用参数为

  • a 查看所有文件
  • l 查看详细信息
  • d 查看目录信息

文件包括以下类型:

  • - 普通文件
  • d 目录文件
  • l 链接文件
  • b 块设备文件
  • c 字符设备文件
  • p 管道文件

每个文件都有所属的用户和组, 文件的基本权限有三个:

  • r 读
  • w 写
  • x 执行

对于一个目录文件来说, 这三项的含义为:

  • r: 可以读取目录内文件列表
  • w: 可以在该目录增删改文件
  • x: 可以进入该目录

权限与数字的对应关系为

  • r – 4
  • w – 2
  • x – 1

所以一个rwx权限可以等价表示为4+2+1=7, 依次类推，可读可写为4+2=6，只读为4

在ls命名列出的文件列表中, 权限从左到右分为三组, 分别是

  1. 文件所有者
  2. 文件所属组
  3. 其他用户

对该文件具有的权限.

修改文件权限指令

chmod [选项] 权限 文件

例如将某个文件修改为所有用户可读可写可执行

用户和用户组

每个用户在创建时都会自动创建一个同名的基本用户组, 该基本用户组只包含该用户.

如果用户以后纳入其他用户组, 则对于该用户来说属于扩展用户组.

一个用户只能有一个基本用户组, 但可以有多个扩展用户组.

Linux Swap

Posted on 2018-05-21

虚拟内存

通过置换功能, 将内存中暂时用不上的数据调至硬盘上, 腾出足够的内存后再将要访问的数据调入内存, 使程序继续执行下去.
这样, 一个大的用户程序能在较小的内存空间中运行, 也可以在内存中同时装入更多的进程使它们并发执行.
在用户感知中, 系统拥有的内存容量比实际内存容量大得多, 但实际上这些多出来的内存是不存在的, 是虚的.
因此将这样的逻辑上容量很大的主存系统称之为虚拟存储器, 即平时所称的虚拟内存.

Swap

Linux中, Swap分区就是划分给系统调出暂时不用的数据的磁盘区域.

在内存不足时, 划分swap分区是十分有必要的, 否则OS必须结束掉某些进程来释放内存空间, 引起进程异常结束.

mkswap file 将指定文件格式化为Swap文件
swapon file 启用指定的Swap文件

URL基础

Posted on 2018-02-18

URL的一般格式

<scheme>://<user>:<password>@<host>:<port>/<path>;<params>?<query>#<frag>

组件 含义 说明
scheme 方案 指定获取资源时使用的协议
user 用户 某些方案访问资源时需要的用户名
password 密码 用户名对应的密码
host 主机 资源宿主服务器的主机名或点分ip地址
port 端口 资源宿主服务器正在监听的端口号, 很多方案都指定了默认的端口号,
path 路径 服务器上资源的本地名
params 参数 某些方案用这个组件来指定输入参数, 形式key=value,分隔符;
query 查询 某些方案用这个组件来指定查询参数, 形式?key=value,分隔符&
frag 片段 一小片或一部分资源的名字(fragment), 仅在本地浏览器生效

方案(scheme)

  1. 大小写无关
  2. 必须以字母作为开头

主机和端口(host&port)

  1. 主机可以用主机名或者ip地址两种形式表示
  2. 在不指定端口时, 根据方案使用默认端口

用户名和密码(user&password)

  1. 这两个组件的需要与否取决于服务器, 例如FTP服务器通常需求这个组件
  2. 在需要这个组件而没有提供时, 通常会插入一个默认的用户和密码

路径(path)

  1. 路径使用/划分为多个路径段, 每个路径段都可以有自己的参数组件

参数(params)

  1. 使用;与其他组件以及互相之间进行分隔
  2. 提供访问资源所需的所有附加信息,

示例：https://example.com/path;type=1;date=2

查询(query)

  1. 使用?与其他组件进行分隔, 使用&互相之间进行分隔.
  2. 查询意在缩小请求资源类型范围.

示例：https://example.com/path?item=mirror&color=bule

片段(frag)

  1. 片段使用#与其他组件进行分隔
  2. 片段仅在html文档使用了锚点时有效
  3. 片段仅仅在本地生效, 被浏览器所处理
  4. 浏览器在请求到整个资源后, 再通过片段来定位到指定的一段

安全字符集&编码机制&字符限制

安全字符集:

各种协议都可能有自己的危险字符, 而作为通用资源定位的URL则必须避开所有的这些字符, 因此, URL能使用的字符集比US-ASCII还要小得多, 能被安全使用的字符集称之为安全字符集

编码机制:

为了能用安全字符集来表示更多的字符, 通过转义表示法来表示不安全的字符, 这样, 出现在URL中的字符都是安全的, 又能表示更多的信息, 这种转义表示法包含一个百分号, 后面跟着两个字符来表示字符ASCII码的十六进制数,

例如, 波浪号~的ASCII码为126(0x7E), 所以波浪号在出现在URL中应当被替换为%7E, 同理, 空格被替换为%20, 百分号本身则被替换为%25

显然, 这种转义表示法对于在ASCII中的所有字符都可以简单地应用(这意味这你可以转义一些本来就很安全的字符, 一样不会引起误解)

但是对于不在ASCII字符集中的字符, 编码的机制就复杂的多, 也没有统一的标准, 但通常会在URL中附加相关信息.

字符限制：

  • 本身含有不安全因素的字符出现在URL中, 必须进行转义
  • 一些保留字符, 如果不是用于保留用途之外的场合时, 必须对其进行转义

例如.这个保留在路径组件中使用的符号, 如果用在URL中而不表示其路径组件中的含义时, 就需要转义,
常见的保留字符包括. .. / # ? ; : $ + @ & = { } | \ ^ ~ [ ] < > "
另外还有一些ASCII中不可打印区间内的字符, URL不可使用

URL快捷方式

URL的表示方式有两种：绝对URL(Absolute URL), 和相对URL(Relative URL)

绝对URL包含访问资源所需的全部信息, 包含URL所有组件

相对URL则只是完整URL的一部分, 想得到完整的URL, 必须相对另一个基础URL(Base URL)进行解析

基础URL的提供方式

  • 使用HTML标记<BASE>显式提供
  • 使用当前URL

基础URL + 相对URL = 绝对URL
